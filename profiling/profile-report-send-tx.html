Sending a transaction is dominated by two serial wallet-service calls that run after the tx is already mined: POST /tx/proposal followed by PUT /tx/proposal/{id}. These together take 1.9–9.4 seconds of the 3.1–11.6 s total. Both requests carry the same signed txHex. Proof-of- work + the tx-mining round-trip is ~900 ms — a minor part of the total.

Headline finding: The POST /tx/proposal + PUT /tx/proposal/{id} round-trip is the hotspot. The client sends the full txHex (~530 B) twice in sequence. If these could be collapsed into a single call, or run in parallel with mining, send time would drop by 1.5–4 s in the typical case.

Anatomy of a sendTransaction() call

Six phases, in order:

  1. prepareTxData (UTXO selection): GET /wallet/tx_outputs to pick inputs for the amount. 310–780 ms — one HTTP call.
  2. signTx (local secp256k1 signing): 19–25 ms — pure CPU, negligible.
  3. runFromMining → MineTransaction.start():
    • submitJob: POST /submit-job to tx-mining-service with the unsigned tx hex. 215–691 ms.
    • handleJobStatus (polled): GET /job-status — interval = max(estimation / 2, 500 ms). Only one poll needed per run here.
    • + an idle wait before the first poll: max(estimation/2, 500 ms) via setTimeout.
  4. handleSendTxProposal — the hotspot:
    • POST /tx/proposal with the txHex. 1023–4531 ms.
    • PUT /tx/proposal/{id} with the same txHex. 849–4881 ms.
Total = HTTP calls (network + backend) + mining wait + tiny CPU. On testnet the network + backend part is ~3 seconds in the best case and multiples of that when the wallet- service tx/proposal endpoint gets slow.

Findings

1. POST + PUT /tx/proposal is the hotspot > 50% of send time

In handleSendTxProposal (src/wallet/sendTransactionWalletService.ts:890–892), the client does: 

const { txProposalId } = await walletApi.createTxProposal(wallet, txHex);  // POST
await walletApi.updateTxProposal(wallet, txProposalId, txHex);              // PUT (same txHex!)

Both requests carry the same signed txHex (~530 B). The POST creates a proposal server-side and returns a UUID; the PUT finalizes it with the same data. Measured durations:

Why are these both so slow? The wallet-service is almost certainly synchronously waiting for fullnode propagation / acceptance during these calls. That's server-side work we can't eliminate from the client — but we can question whether we need two round-trips or whether they can overlap with mining.

2. UTXO selection takes 310–780 ms for one HTTP call

GET /wallet/tx_outputs returns ~345 B of JSON describing the available UTXOs. Even warm, this takes 310 ms on testnet. Cold first-call of the process takes 780 ms (same TLS/auth cold-start we saw in the load profile). Consistent part of the hot path, hard to eliminate without a client-side UTXO cache.

3. Mining (PoW) is NOT the bottleneck

The tx-mining service returns in ~900 ms total (submit 215–691 ms, one status poll at 215–220 ms, plus a ~500 ms idle wait between them). Everyone blames PoW for slow sends. On testnet it's only ~8–15% of the total.

4. signTx is cheap (19–25 ms)

secp256k1 signing of a single-input single-output transaction is fast even in pure JS (bitcore-lib). Not worth optimizing.

5. An idle setTimeout wait before the first mining-status poll

handleJobStatus in src/wallet/mineTransaction.ts:147 computes poll_time = max(estimation / 2, MIN_POLLING_INTERVAL) * 1000 and sleeps before the first status GET. With MIN_POLLING_INTERVAL = 0.5 (0.5 s) and testnet estimations typically ~1 s, this adds a flat ~500 ms of pure idle after the tx was submitted. If mining finishes earlier, we still wait.

Recommendations

  1. Ask the wallet-service team why POST + PUT are both needed. Can the POST body include finalize: true so a single call does both? Or can the PUT start immediately after POST returns without server-side re-validation? Any reduction in either call's latency maps 1:1 to user-visible savings. This is the single biggest lever.
  2. Overlap POST /tx/proposal with mining. The POST could potentially accept an un-mined tx (or a "pre-proposal" stub) and validate inputs / UTXOs in parallel with the tx-mining PoW. Then the PUT updates it with nonce+parents+timestamp once mining finishes. Net savings: up to 900 ms per send.
  3. Drop the blind 500 ms setTimeout before first poll. Issue the first GET /job-status immediately after POST /submit-job returns, or use a push/stream channel from tx-mining if available. Net: ~500 ms cheaper per send.
  4. Short-circuit UTXO selection for trivial sends — if the wallet already knows about a UTXO >= requested amount from a previous query or WS update, skip the GET /wallet/tx_outputs. Net: ~300–780 ms.

What was not measured

  • Transactions with many inputs/outputs (multi-token, batch sends). signTx scales linearly — with 10+ inputs the local CPU time grows, and tx_outputs response size grows too. Worth a follow-up.
  • Fullnode-directed sends (non-wallet-service path). The old HathorWallet facade uses a different proposal pipeline.
  • Nano contract transactions go through the same handleSendTxProposal hotspot — likely the same findings apply, plus extra server-side work.

Methodology

Harness: profiling/profile-send-tx.ts. It:

  • starts the wallet normally (not profiled — see the load-profile report for that);
  • monkey-patches SendTransactionWalletService.prototype (prepareTx, prepareTxData, selectUtxosToUse, signTx, runFromMining, handleSendTxProposal) and MineTransaction.prototype (submitJob, handleJobStatus) to emit perf_hooks marks;
  • installs axios interceptors that capture every HTTP call made to wallet-service and tx-mining-service (method, URL, status, duration, bytes);
  • calls wallet.sendTransaction(addr, 1n, { pinCode }) with the wallet's own address 0 as destination (self-send, no economic effect);
  • writes per-run JSON to profiling/profiles/sendtx-<label>.json.

Reproducing

RUNS=2 SEND_AMOUNT=1 NETWORK=testnet npm run profile:send-tx
