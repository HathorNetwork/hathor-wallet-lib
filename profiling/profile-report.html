Read-only buys you a modest 50–190 ms on state=Ready — mostly from skipping PBKDF2 on the seed. The big structural costs (HTTPS latency to wallet-service, the sequential WebSocket handshake that runs after state=Ready) are unchanged. Read-only does not fix the slow-load complaint.

The single biggest opportunity in either mode: start the WebSocket handshake in parallel with HTTP, not after onWalletReady(). That alone saves ~0.8 s off cold time-to-fully-ready. Applies to both modes equally.

Findings

1. The WebSocket handshake runs after HTTP, not in parallel — in both modes

In onWalletReady(), the client awaits getNewAddresses(true), then calls setupConnection(), then sets state = READY. The state=Ready event fires before the WS has even opened — clients that rely on the WS for live balance updates wait another ~1 s on warm and up to ~8 s on a bad cold run.

Raw WS events from all 6 runs (time relative to setupConnection() call):

Full (seed):
  Run 1 (fresh): SEND join @ 853 ms   RECV join-success @ 1418 ms   (WS ready: 1.42 s)
  Run 2 (warm):  SEND join @ 781 ms   RECV join-success @ 1167 ms   (WS ready: 1.17 s)
  Run 3 (warm):  SEND join @ 799 ms   RECV join-success @ 1225 ms   (WS ready: 1.23 s)

Read-only (xpub):
  Run 1 (fresh): SEND join @ 3785 ms  RECV join-success @ 8112 ms   (WS ready: 8.11 s)   ← pathological
  Run 2 (warm):  SEND join @ 1113 ms  RECV join-success @ 1463 ms   (WS ready: 1.46 s)
  Run 3 (warm):  SEND join @  759 ms  RECV join-success @ 1109 ms   (WS ready: 1.11 s)

The ~800 ms before SEND is TCP + TLS + WS-upgrade to the ws.wallet-service.testnet.hathor.network host — a different hostname from the HTTP service, so every wallet load re-handshakes a fresh TLS session here regardless of keep-alive. The remaining latency is the server's join ack.

Pathological run: RO Run 1 took 3.8 s to open the WS, then 4.3 s for the server to ack join, plus a heartbeat ping fired mid-wait at 5 s. We've seen similar one-off 5+ s loads in full mode too — testnet WS edge has very bad tail latency and DEFAULT_JOIN_TIMEOUT = 5000 masks nothing.

2. Read-only is barely faster than full mode

Read-only eliminates:

  • The 133 ms cold PBKDF2 (generateAccessDataFromSeed → now just generateAccessDataFromXpub, ~6–18 ms);
  • The POST /wallet/init call.

But /wallet/init was already running in parallel with /auth/token in full mode, so removing it doesn't help wall-clock time much. Only the PBKDF2 saves anything on the critical path. Net: −50 to −190 ms depending on cache state.

Put differently: if your read-only consumers are asking “why is read-only still slow” — it's because they hit the same ~260 ms auth POST, the same ~280 ms addresses GET, and the same ~1.2 s WebSocket as full mode.

3. POST /auth/token/readonly is NOT faster than POST /auth/token

Same hostname, same TLS path, same approximate size. Cold ~1030 ms, warm ~260 ms. The read-only endpoint only saves the client-side xpubkey signature computation (negligible) — server-side latency is identical.

4. GET /wallet/addresses/new is serialized and always ~280 ms

It can't start until we have a confirmed walletId. In read-only mode, you can pass startReadOnly({ skipAddressFetch: true }) to skip it entirely — that'd shave ~280 ms off every load if you don't need gap-limit addresses pre-populated.

5. Suspects that didn't pay out

  • WALLET_STATUS_POLLING_INTERVAL = 1000 ms: never fires in any of these runs because the wallet is already loaded server-side. Only matters on a first-ever wallet create (not measured here).
  • validateAndRenewAuthToken awaited per authenticated request: the JWT caches, subsequent calls measure at 0.1 ms.

Recommendations

  1. Start WebSocket in parallel with HTTP. Kick setupConnection() as soon as we have walletId (right after the xpub is computed, before the /wallet/init or /auth/token/readonly await). Expected savings: ~0.8–1.4 s on cold fully-ready time. Applies to both modes.
  2. Pass skipAddressFetch: true to startReadOnly for read-only consumers that don't need gap addresses. Expected savings: ~280 ms on every load. (Not applicable to full mode — full mode fetches addresses implicitly.)
  3. Skip /wallet/init on returning full-mode wallets. If we have a cached walletId, go straight to GET /wallet/status; only fall back to /wallet/init if status is 404. Expected savings: ~250 ms warm. (Not applicable to read-only — it already skips /wallet/init.)
  4. Shorten DEFAULT_JOIN_TIMEOUT from 5000 ms to ~1500 ms, so the pathological WS-handshake case doesn't hold the user hostage for 8+ s.
  5. Ask the wallet-service team about HTTP/2 and WS edge latency. Cold POST /auth/token* consistently takes ~1 s — that's infrastructure-level, not client-fixable. Tail latency on the WS edge (3.8 s to open in RO Run 1) is even more concerning.

What was not measured

First-ever wallet creation (server has never seen this seed): /wallet/init returns 200 creating and start() enters pollForWalletStatus(), which polls with setInterval(..., 1000) — the first check fires at t=1 s, not t=0. If users complain about slowness on brand-new wallets specifically, that's almost certainly the culprit. Easy follow-up with a fresh seed.

Methodology

Harness at profiling/profile-wallet-load.ts. It:

  • loads .env.profile (gitignored) for the wallet seed;
  • for read-only mode: derives the account-path xpub from the same seed via getXPrivKeyFromSeed + deriveNonCompliantChild(P2PKH_ACCT_PATH), constructs the wallet with xpub instead of seed, and calls startReadOnly();
  • patches axios.create with interceptors that record {method, url, status, startTime, duration, req/resBytes};
  • monkey-patches walletUtils.generateAccessDataFromSeed / generateAccessDataFromXpub and several instance methods (validateAndRenewAuthToken, renewAuthToken, getReadOnlyAuthToken, generateCreateWalletAuthData, pollForWalletStatus, onWalletReady, getNewAddresses, setupConnection) to emit perf_hooks marks/measures;
  • hooks the internal WS client's sendMessage and join-success event to capture raw handshake timing;
  • writes per-run JSON to profiling/profiles/run-<label>.json.

Mark names are namespaced with a per-run prefix to avoid cross-run pollution.

Reproducing

# Full (seed) mode:
SCENARIO=both RUNS=3 ENABLE_WS=1 WAIT_WS_READY=1 NETWORK=testnet \
  npm run profile:wallet

# Read-only (xpub derived from same seed) mode:
READONLY=1 SCENARIO=both RUNS=3 ENABLE_WS=1 WAIT_WS_READY=1 NETWORK=testnet \
  npm run profile:wallet
